## Overview

[Windsurf](https://windsurf.com/) is an AI-native code editor built on the VS Code architecture. Originally developed by **Codeium** (founded in 2021 as Exafunction by MIT alumni Varun Mohan and Douglas Chen), Windsurf launched as a standalone IDE in late 2024. In mid-2025, OpenAI attempted a $3 billion acquisition that ultimately fell through, after which Google executed a talent acquisition of key leadership, and **Cognition AI** (makers of Devin) acquired Windsurf's product, IP, and remaining team in December 2025 for approximately $250 million.

Windsurf's core differentiator is **Cascade**, a persistent AI agent that deeply indexes your codebase, maintains real-time awareness of your actions in the editor, and executes multi-step plans including file creation, code editing, and terminal command execution. The editor supports the [Model Context Protocol (MCP)](https://modelcontextprotocol.io/) for connecting to external tools and data sources.

<Info>
  Windsurf supports multiple AI models including its own SWE model family, Anthropic Claude, OpenAI GPT, and Google Gemini. Users on Free and Pro plans can also bring their own API keys (BYOK) for direct model access.
</Info>

### Key Features

| Feature | Description |
|---|---|
| **Cascade Agent** | Persistent AI agent with deep codebase understanding, multi-file editing, terminal execution, and real-time awareness of developer actions |
| **Write and Chat Modes** | Write mode for autonomous code generation and modification; Chat mode for questions and guidance |
| **Turbo Mode** | Autonomous terminal command execution without manual approval (configurable deny list) |
| **MCP Integration** | Connect to external tools, databases, APIs, and services via Model Context Protocol servers |
| **Tab / Supercomplete** | Intelligent code completions powered by the SWE-1-mini model, optimized for real-time latency |
| **Live Preview** | In-editor website preview with click-to-edit element selection |
| **Memory System** | Persistent knowledge layer that learns coding style, patterns, and project conventions across sessions |
| **Rules** | Project-level and global instruction files that guide Cascade's behavior |
| **Arena Mode** | Side-by-side model comparison with blind voting to discover which models work best for your workflow |
| **Cascade Hooks** | Custom commands executed at key points during Cascade's workflow (Enterprise) |

### Rules System

Windsurf uses markdown-based rule files to provide custom instructions to Cascade. Rules are the highest-impact configuration for shaping Cascade's behavior.

<CodeGroup>

```markdown .windsurf/rules/general.md
---
trigger: always
---

This is a TypeScript monorepo using pnpm workspaces.

Tech stack:
- Frontend: React 19 with Next.js 15
- Backend: Node.js with Express
- Database: PostgreSQL with Drizzle ORM
- Testing: Vitest for unit tests, Playwright for E2E

Coding conventions:
- Use strict TypeScript with no `any` types
- Prefer functional patterns over classes
- All public APIs must have JSDoc comments
- Use named exports, not default exports
```

```markdown .windsurf/rules/testing.md
---
trigger: glob
globs: **/*.test.ts, **/*.spec.ts
---

All test files must:
- Use `describe`/`it` blocks
- Mock external API calls
- Include both happy path and error cases
- Use factory functions for test data, not raw objects
```

```markdown .windsurf/rules/api.md
---
trigger: glob
globs: src/api/**/*.ts
---

API route conventions:
- Use Zod for request/response validation
- Return consistent error response shapes
- Include rate limiting middleware on all public endpoints
- Log all errors with structured JSON logging
```

</CodeGroup>

### Rule Levels

| Level | Location | Scope |
|---|---|---|
| **Global** | `~/.windsurf/rules/` | Applied to every project opened in Windsurf |
| **Workspace** | `<project>/.windsurf/rules/` | Project-specific rules, can use glob triggers |
| **System** | Managed by admin | Organization-wide baseline standards (Enterprise) |
| **Legacy** | `<project>/.windsurfrules` | Single-file format, still supported |

### Rule Trigger Types

| Trigger | Description |
|---|---|
| `always` | Applied to every Cascade interaction |
| `glob` | Applied only when working with files matching the specified glob pattern |

<Warning>
  Individual rule files are limited to 6,000 characters each. The total combined characters for global and workspace rules must not exceed 12,000. If this limit is exceeded, global rules take priority, followed by workspace rules, with any excess truncated.
</Warning>

### Rule Discovery

Windsurf automatically discovers rules from multiple locations: the current workspace and sub-directories (all `.windsurf/rules` directories), the git repository structure (searching up to the git root), and when multiple folders are open, rules are deduplicated.

<Note>
  If `.windsurf/rules` is listed in `.gitignore`, the rules will not be loaded. This is a known limitation.
</Note>

### The .codeiumignore File

The `.codeiumignore` file uses `.gitignore` syntax to exclude files from Cascade's indexing and context. Ignored files will not be indexed and do not count against workspace size limits.

```text .codeiumignore
# Exclude build artifacts
dist/
build/
node_modules/

# Exclude generated files
*.generated.ts
*.min.js

# Exclude sensitive files
.env*
secrets/
```

## MCP Servers

Windsurf supports the Model Context Protocol for connecting Cascade to external tools, data sources, and services. MCP servers are configured in `mcp_config.json`.

### Configuration Format

MCP servers are defined in `~/.codeium/windsurf/mcp_config.json` using a JSON format with three supported transport types: **stdio**, **Streamable HTTP**, and **SSE**.

<CodeGroup>

```json stdio (Local Server)
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_PERSONAL_ACCESS_TOKEN": "${env:GITHUB_TOKEN}"
      }
    },
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/path/to/workspace"],
      "disabled": false
    }
  }
}
```

```json Streamable HTTP (Remote Server)
{
  "mcpServers": {
    "remote-api": {
      "serverUrl": "https://your-server.com/mcp",
      "headers": {
        "Authorization": "Bearer ${env:API_TOKEN}"
      }
    }
  }
}
```

```json SSE (Server-Sent Events)
{
  "mcpServers": {
    "streaming-server": {
      "serverUrl": "https://mcp.example.com/sse"
    }
  }
}
```

</CodeGroup>

### Environment Variable Interpolation

The `mcp_config.json` file supports environment variable interpolation using `${env:VARIABLE_NAME}` syntax in the `command`, `args`, `env`, `serverUrl`, `url`, and `headers` fields.

### Adding MCP Servers

<Tabs>
  <Tab title="MCP Marketplace">
    Click the MCPs icon in the top-right corner of the Cascade panel. Browse the built-in marketplace to discover and enable available MCP servers.
  </Tab>
  <Tab title="Settings UI">
    Navigate to **Windsurf Settings > Cascade > MCP Servers**. Add servers through the configuration form, or click **View raw config** to edit `mcp_config.json` directly.
  </Tab>
  <Tab title="Manual (mcp_config.json)">
    Edit `~/.codeium/windsurf/mcp_config.json` directly. The file is created automatically when you add your first server through the UI, or you can create it manually.
  </Tab>
</Tabs>

### OAuth Support

Windsurf supports OAuth for all three transport types (stdio, Streamable HTTP, SSE), enabling authenticated access to MCP servers that require OAuth-based authorization.

### Tool Limits

Cascade has a limit of **100 total tools** across all enabled MCP servers at any given time. You can toggle individual tools on or off from each MCP server's settings page to stay within this limit.

<Note>
  When `mcp_config.json` is modified, only the affected MCP server instance is initialized or refreshed. Other server instances remain unaffected.
</Note>

## Cascade Modes and Capabilities

### Interaction Modes

| Mode | Description | Automation Level |
|---|---|---|
| **Write** | Cascade creates, modifies, and deletes files; runs terminal commands; iterates on errors automatically | ~90% automated |
| **Chat** | Cascade answers questions, explains code, and generates snippets for manual application | ~50% automated |

Write mode is enabled by default. Switch between modes in the Cascade panel.

### Command Auto-Execution Levels

| Level | Behavior |
|---|---|
| **Off** | All terminal commands require manual approval |
| **Auto** | Safe commands execute automatically; risky commands require approval |
| **Turbo** | All commands execute automatically unless they appear on the deny list |

<Warning>
  Enable Turbo mode only on trusted codebases. Enterprise administrators can set a maximum allowed auto-execution level and configure organization-wide allow and deny lists.
</Warning>

### Built-in Capabilities

Cascade has the following built-in capabilities without any additional configuration:

| Capability | Description |
|---|---|
| **Multi-file editing** | Coordinated changes across multiple files simultaneously |
| **Terminal execution** | Run shell commands, install packages, execute tests, start dev servers |
| **Auto-fix errors** | Automatically fix linter errors in generated code |
| **Codebase indexing** | Deep indexing of project structure, dependencies, and patterns |
| **Real-time awareness** | Monitors your manual edits and actions in the editor |
| **Memory persistence** | Learns coding style and patterns across sessions |
| **Context retrieval** | Intelligent context retrieval using the swe-grep model |

### Tool Call Limits

Cascade can make up to **25 tool calls per prompt**. If the execution stops before completion, type "continue" to resume from where it left off (each continuation counts as a new prompt).

### Cascade Hooks (Enterprise)

Hooks allow custom commands to execute at key points during Cascade's workflow:

| Hook Type | Description |
|---|---|
| **On model response** | Execute commands when Cascade generates a response (e.g., audit logging) |
| **On user prompt** | Execute commands when a user submits a prompt (e.g., policy enforcement, blocking violating prompts) |

## Model Configuration

### In-House Models (SWE Family)

| Model | Purpose |
|---|---|
| **SWE-1** | First agentic coding model, used for Cascade agent tasks |
| **SWE-1.5** | Latest frontier model with near-SOTA performance |
| **SWE-1-mini** | Powers Tab / Supercomplete, optimized for real-time latency |
| **swe-grep** | Powers context retrieval and Fast Context |

### Supported Third-Party Models

| Provider | Models |
|---|---|
| **OpenAI** | GPT-5 (low/medium/high reasoning), GPT-5.1, GPT-5.1-Codex, GPT-5.2-Codex |
| **Anthropic** | Claude Opus 4.5, Sonnet 4.5, Claude 3.7 Sonnet (with Thinking support) |
| **Google** | Gemini 3 Flash |

### Model Selection

Select models from the dropdown in the Cascade panel (accessible via `Cmd+L` / `Ctrl+L`). Each model consumes a different number of prompt credits per interaction.

### Bring Your Own Key (BYOK)

Individual users on Free and Pro plans can provide their own API keys for direct model access. Models supporting BYOK are labeled in the model dropdown. Configure API keys in the subscription settings page.

## Installation

<Tabs>
  <Tab title="macOS">
    Download the `.dmg` installer from [windsurf.com/download/editor](https://windsurf.com/download/editor). Open the `.dmg` file and drag Windsurf to the Applications folder. On Apple Silicon Macs, Windsurf runs natively without Rosetta.

    **System requirements:** macOS versions with Apple security update support (typically the latest release and two previous versions). macOS 10.15 is not supported.
  </Tab>
  <Tab title="Windows">
    Download the `.exe` installer from [windsurf.com/download/editor](https://windsurf.com/download/editor). Run the installer and follow the setup wizard. Default installation path: `C:\Program Files\Windsurf`.

    **System requirements:** Windows 10 (64-bit). Available for x64 and arm64.
  </Tab>
  <Tab title="Linux">
    Download the `.deb` (Debian/Ubuntu) or `.rpm` (Fedora) package from [windsurf.com/download/editor](https://windsurf.com/download/editor).

    ```bash
    # Debian / Ubuntu
    sudo dpkg -i windsurf_*.deb

    # Fedora / RHEL
    sudo rpm -i windsurf_*.rpm
    ```

    A `.tar.gz` archive is also available for manual installation.

    **System requirements:** glibc >= 2.28, glibcxx >= 3.4.25 (e.g., Ubuntu 20+, Debian 10+, Fedora 36+, RHEL 8+).
  </Tab>
</Tabs>

### Post-Installation

- **Import VS Code settings:** Use the command palette (`Cmd+Shift+P` / `Ctrl+Shift+P`) to import settings, extensions, keybindings, and themes from VS Code or Cursor.
- **Incompatible extensions:** Other AI code completion extensions (e.g., GitHub Copilot) are incompatible with Windsurf and should be disabled.
- **Updates:** Windsurf updates automatically. Click "Restart to Update" when prompted.

### Windsurf Next (Pre-release)

Windsurf Next is a pre-release channel for early access to new features. Features typically ship to Windsurf Next first and then roll into the stable release. Download separately from the Windsurf website.

### Installing MCP Server Extensions

To programmatically register an MCP server, merge the server definition into the `mcpServers` object in `mcp_config.json`:

```json
{
  "mcpServers": {
    "package-server-name": {
      "command": "npx",
      "args": ["-y", "@scope/mcp-server-package"],
      "env": {
        "API_KEY": "${env:SOME_API_KEY}"
      }
    }
  }
}
```

**Key considerations:**

- The `mcpServers` object is a flat map; each key is a unique server identifier
- Merge carefully -- do not overwrite existing server entries
- The config file may not exist yet; create it with the full structure if absent
- The file supports three transport types: `stdio` (uses `command`/`args`), Streamable HTTP (uses `serverUrl`), and SSE (uses `serverUrl`)
- Use `${env:VAR_NAME}` syntax for environment variable interpolation instead of hardcoding secrets
- Cascade has a limit of 100 total tools across all MCP servers; be mindful when adding servers with many tools
- When the config file is modified, only the affected MCP server is refreshed (no full restart required)
- Set `"disabled": true` on a server entry to register it without enabling it

### Installing Project Instructions

To install project-level instructions, create markdown files in the `<project>/.windsurf/rules/` directory:

1. Create the `.windsurf/rules/` directory in the project root if it does not exist
2. Write rule files as markdown with optional YAML frontmatter for trigger configuration
3. Each rule file should focus on a single topic (e.g., `general.md`, `testing.md`, `api.md`)

For the legacy format, write a `.windsurfrules` file to the project root. If one already exists, the installer should append to it.

```markdown
---
trigger: always
---

Your project-specific instructions here.
```

<Warning>
  Individual rule files are limited to 6,000 characters. The combined total for global and workspace rules must not exceed 12,000 characters. Keep rules concise and focused.
</Warning>

### Config File Format

The MCP configuration uses standard JSON. The installer must:

1. Check if `~/.codeium/windsurf/mcp_config.json` exists; create the directory and file if absent
2. Parse the existing JSON file
3. Deep-merge new server entries into the `mcpServers` object, preserving existing entries
4. Write the result back as formatted JSON

Editor settings use the standard VS Code `settings.json` format and location.

### Auto-Execution Modes

An installer should be aware of Windsurf's command auto-execution levels, which control whether Cascade can run terminal commands without user approval:

| Level | Behavior |
|---|---|
| `Off` | All commands require manual approval |
| `Auto` | Safe commands run automatically; risky commands require approval |
| `Turbo` | All commands run automatically unless on the deny list |

### Enabling MCP

MCP must be enabled in Windsurf settings before servers will be active. The setting is found under **Settings > Advanced Settings > Cascade > Model Context Protocol (MCP)**. Programmatically, this can be toggled in `settings.json`.
