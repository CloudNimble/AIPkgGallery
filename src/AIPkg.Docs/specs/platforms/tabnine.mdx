## Overview

[Tabnine](https://www.tabnine.com/) is a privacy-focused AI code assistant developed by Tabnine (formerly Codota). It provides code completions, AI chat, and autonomous coding agents across VS Code, JetBrains IDEs, Eclipse, and Visual Studio 2022/2026 [^1]. Tabnine supports extensibility through MCP servers, custom guidelines, custom commands, and a standalone CLI agent. It was named a Visionary in the September 2025 Gartner Magic Quadrant for AI Code Assistants [^2].

Key extensibility points:

| Feature | Support |
|---|---|
| MCP Servers | Yes, via `.tabnine/mcp_servers.json` [^3] |
| Custom Instructions (Guidelines) | Yes, via `.tabnine/guidelines/*.md` [^4] |
| Custom Commands | Yes, via `.tabnine/agent/commands/` [^5] |
| CLI Agent | Yes, `tabnine` CLI [^6] |
| Model Flexibility | Yes, switchable LLMs including GPT-4o, Claude, Gemini, Llama, and private endpoints [^2] |

## MCP Server Integration

Tabnine supports the Model Context Protocol (MCP), allowing agents to connect to external tools and data sources through a standardized interface [^3]. MCP servers are configured via JSON files in the `.tabnine/` directory.

### Configuration File

MCP servers are defined in `mcp_servers.json`, which can be placed at two levels:

| Scope | Path |
|---|---|
| Project-level | `<project>/.tabnine/mcp_servers.json` |
| Global (user-level) | `~/.tabnine/mcp_servers.json` |

### Transport Types

Tabnine auto-detects the transport type based on the configuration fields [^3] [^13]:

| Transport | Detection | Use Case |
|---|---|---|
| STDIO | `command` field present | Local servers, CLI tools |
| Streamable HTTP | `url` field present | Modern remote APIs |
| SSE (Legacy) | `transport: "sse"` explicitly set | Legacy remote servers |

### Configuration Fields

| Field | Required | Description |
|---|---|---|
| `command` | For STDIO | Executable to launch the server |
| `args` | No | Command-line arguments array |
| `env` | No | Environment variables object |
| `cwd` | No | Working directory for the process |
| `url` | For HTTP/SSE | Remote server endpoint URL |
| `requestInit` | No | Fetch API RequestInit object (headers, auth) |
| `transport` | For SSE only | Must be `"sse"` for legacy servers |
| `sessionId` | No | Session identifier for remote connections |
| `eventSourceInit` | No | EventSource-specific options |

<CodeGroup>
```json Local STDIO Server
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/path/to/project"],
      "env": {},
      "cwd": "/path/to/project"
    }
  }
}
```

```json Remote Server with JWT
{
  "mcpServers": {
    "remote-api": {
      "url": "https://api.example.com/mcp",
      "requestInit": {
        "headers": {
          "Authorization": "Bearer eyJhbGciOi..."
        }
      }
    }
  }
}
```

```json Legacy SSE Server
{
  "mcpServers": {
    "legacy-sse": {
      "transport": "sse",
      "url": "https://legacy.example.com/sse",
      "requestInit": {
        "headers": {
          "Authorization": "Bearer token"
        }
      },
      "eventSourceInit": {}
    }
  }
}
```
</CodeGroup>

<Note>
  OAuth credential caches are stored in the `~/.mcp_auth` directory. If tokens are revoked externally, you may need to manually delete this folder to clear cached credentials [^3].
</Note>

<Warning>
  MCP features require the Tabnine Agentic Platform tier ($59/user/month). The Code Assistant tier does not include MCP support [^14].
</Warning>

## Custom Instructions (Guidelines)

Tabnine uses Markdown-based guidelines to customize AI agent behavior [^4]. Guidelines are natural-language instructions placed in `.tabnine/guidelines/` directories.

### File Format

Guidelines are plain Markdown files (`.md`) with no required frontmatter. Tabnine recommends organizing content hierarchically and keeping files to 500 lines or less [^4].

### Guideline Scopes

| Scope | Path |
|---|---|
| Project-level | `<project>/.tabnine/guidelines/*.md` |
| Global (user-level) | `~/.tabnine/guidelines/*.md` |

### Supported Guideline Types

Guidelines can define [^4]:

- **Custom System Prompts** -- How the agent should behave
- **Workflow Rules** -- Procedures and processes to follow
- **Tool Instructions** -- When and how to use specific tools
- **Team Standards** -- Coding practices and conventions

### Enterprise Team Designation

Enterprise administrators can assign guidelines to all teams or to specific teams. Organization-wide guidelines set in the Admin UI take precedence over local files [^4].

```markdown .tabnine/guidelines/api-standards.md
# API Standards

## Naming Conventions
- Use camelCase for function and variable names
- Use PascalCase for class and interface names
- Prefix interfaces with "I" (e.g., IUserService)

## Error Handling
- Always use custom error classes extending BaseError
- Log errors with structured metadata
- Return appropriate HTTP status codes

## Examples

### Good
- `getUserById(id: string): Promise<User>`
- `class UserNotFoundError extends BaseError {}`

### Bad
- `get_user(id)` (wrong casing, no types)
- `throw new Error("not found")` (generic error)
```

## Context Files

Tabnine supports `TABNINE.md` files as persistent, hierarchical context files that provide ongoing project awareness to the AI model [^5]. Place a `TABNINE.md` file in your project root or any subdirectory to supply relevant context.

You can also use `@<file/folder>` syntax in chat to direct the AI's attention to specific files or folders [^5].

## CLI Agent

Tabnine provides a CLI agent (`tabnine`) that runs natively in the terminal as a fully autonomous or semi-autonomous AI coding agent [^6].

### Installation

<Tabs>
  <Tab title="Linux / macOS">
    ```bash
    export TABNINE_HOST="https://console.tabnine.com"
    curl $TABNINE_HOST/update/cli/installer.mjs | node --input-type=module - $TABNINE_HOST
    ```
    The binary is installed to `~/.local/bin/tabnine`. Ensure this directory is in your `PATH` [^6].
  </Tab>
  <Tab title="Windows (PowerShell)">
    ```powershell
    $env:TABNINE_HOST = "https://console.tabnine.com"
    irm $env:TABNINE_HOST/update/cli/installer.mjs | node --input-type=module - $env:TABNINE_HOST
    ```
  </Tab>
</Tabs>

### Key CLI Commands

| Command | Purpose |
|---|---|
| `tabnine -p "prompt"` | Non-interactive single prompt execution [^6] |
| `tabnine -p "prompt" --output-format json` | JSON-formatted output [^6] |
| `cat file.js \| tabnine -p "Review this"` | Pipe input to the agent [^6] |
| `/model` | Switch between AI models [^5] |
| `/settings` | Adjust CLI behavior [^5] |
| `/directory add/show` | Manage project workspace folders [^5] |
| `/help` | Show all available commands [^6] |
| `/bug` | Report issues [^5] |
| `Ctrl+Y` | Toggle YOLO mode (auto-execution without confirmation) [^5] |
| `!command` | Run shell commands directly (e.g., `!npm run start`) [^5] |

### Agent Tool Permissions

The Tabnine agent provides nine built-in tools with configurable permission levels (auto-approve, ask first, or disable) [^15]:

| Tool | Function |
|---|---|
| Read Project Files | Access code and documentation for context |
| Create Project Files | Generate new files and directories |
| Apply Code | Modify or write to existing files |
| Read Terminal | Review command outputs |
| Run Command | Execute terminal commands |
| List Directory | Display folder contents |
| Get Diagnostics | Gather environment and dependency info |
| Local Workspace Search | Find patterns in workspace |
| Search File Content | Code-aware search |

## Enterprise Features

### Deployment Options

Tabnine Enterprise supports multiple deployment models [^16]:

| Deployment | Description |
|---|---|
| SaaS | Cloud-hosted by Tabnine |
| VPC (Single-Tenant) | Deployed in your AWS, GCP, or Azure VPC |
| On-Premises | Kubernetes cluster on your own servers |
| Air-Gapped | Fully offline with no external network access |

### Enterprise Context Engine

The Enterprise Context Engine builds and maintains a continuously updated representation of an organization's software environment, enabling AI agents to work with current understanding of systems rather than static snapshots [^17]. It connects to repositories on Bitbucket, GitHub, GitLab, and Perforce P4 [^14].

### Model Flexibility

Enterprise administrators can [^2] [^18]:

- Switch between built-in models (GPT-4o, Claude, Gemini, Llama, Mistral, and others)
- Register private LLM endpoints and make them available per project
- Connect to models hosted in AWS VPC, Azure, OCI, or on-premises infrastructure

### Installing MCP Server Extensions

To programmatically register an MCP server, merge a server entry into the `mcp_servers.json` file at either the project or global level:

```json
{
  "mcpServers": {
    "package-server-name": {
      "command": "npx",
      "args": ["-y", "@scope/mcp-server-package"],
      "env": {
        "SOME_API_KEY": "value"
      }
    }
  }
}
```

**Key considerations:**
- The `mcpServers` field is an **object map** keyed by server name [^3]
- When merging, preserve all existing server entries
- The same JSON format is used across VS Code, JetBrains, and CLI -- the file is always `.tabnine/mcp_servers.json` [^3]
- The file can be placed at project level or in the user's home directory

<Note>
  Unlike some other platforms, Tabnine uses a single, consistent MCP configuration format (`mcp_servers.json`) across all IDEs and the CLI. The installer does not need to handle different formats per IDE.
</Note>

### Installing Custom Instructions

To install project-level custom instructions, create a Markdown file in the `.tabnine/guidelines/` directory:

```bash
mkdir -p .tabnine/guidelines
cat > .tabnine/guidelines/my-package-guidelines.md << 'EOF'
# My Package Guidelines

These instructions guide the AI agent when working with this codebase.

## Coding Standards
- Follow the established patterns in this repository
- Write tests for all new functions
EOF
```

**Key considerations:**
- Guidelines are plain Markdown files with no required frontmatter [^4]
- Multiple guideline files can coexist in the directory
- Keep files to 500 lines or less for optimal performance [^4]
- Guidelines use natural language -- no special syntax required

### Config File Format

Tabnine uses JSON for MCP server configuration and Markdown for guidelines:

- **MCP servers**: `mcp_servers.json` is standard JSON. Parse existing content, deep-merge new server entries into the `mcpServers` object, and write back preserving existing entries.
- **Guidelines**: Plain Markdown files. Create new files rather than modifying existing ones to avoid conflicts.
- **VS Code settings**: Standard VS Code `settings.json` for extension-specific settings (e.g., `tabnine.disableLineRegex`).

### Non-Interactive Considerations

For CI/CD or automated workflows:

```bash
# Non-interactive single prompt
tabnine -p "Run the test suite and report results"

# JSON output for parsing
tabnine -p "List all TODO comments" --output-format json

# Pipe file content for review
cat src/main.ts | tabnine -p "Review this code for security issues"
```

The Tabnine CLI also provides a GitHub Actions integration for automating code review on pull requests [^19].

<Warning>
  The Tabnine CLI requires Node.js to be installed for the initial setup. It also requires Tabnine Agents to be enabled for your team. Authentication is handled via the CLI login flow -- there are no user-provided API keys for model access [^6].
</Warning>
