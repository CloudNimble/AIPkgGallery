## Overview

[Trae](https://www.trae.ai/) (The Real AI Engineer) is an AI-native code editor built by **ByteDance**, based on the VS Code architecture.[^1] Extensibility is via MCP servers (`.trae/mcp.json`), **Rules** (markdown files in `.trae/rules/`), **Skills** (`SKILL.md`-based capability sets in `.trae/skills/`), and **Custom Agents** (user-defined agent configurations with prompt, tool, and MCP scoping).[^2][^3][^4]

<Note>
  Trae exists in two variants: **Trae** (international, at [trae.ai](https://www.trae.ai/)) and **Trae CN** (Chinese domestic, formerly known as MarsCode). Both share the same extensibility surface and project-level configuration structure. The primary differences are available AI models (Claude/GPT for international; Doubao/DeepSeek for CN) and the global configuration directory path.[^5][^6]
</Note>

## Configuration

### File Locations

Trae uses JSON configuration files and Markdown-based rule/skill files. Configuration is split between global (user-level) and project-level locations.

<Tabs>
  <Tab title="Trae (International)">
    | File | Path | Purpose |
    |---|---|---|
    | Project MCP config | `<project>/.trae/mcp.json` | MCP servers scoped to a specific project[^2] |
    | Project rules | `<project>/.trae/rules/*.md` | Markdown rule files for AI behavior[^3] |
    | Project skills | `<project>/.trae/skills/` | Installed skill directories containing `SKILL.md` files[^4] |
    | Skill config | `<project>/.trae/skill-config.json` | Tracks disabled project skills[^4] |
    | AI ignore | `<project>/.trae/.ignore` | Files/folders to exclude from AI context[^7] |
    | Global skills | `~/.trae/skills/` | Skills available across all projects[^8] |
    | User rules | Settings UI > Rules | Personal rules applied to all projects[^3] |
  </Tab>
  <Tab title="Trae CN">
    | File | Path | Purpose |
    |---|---|---|
    | Project MCP config | `<project>/.trae/mcp.json` | MCP servers scoped to a specific project[^2] |
    | Project rules | `<project>/.trae/rules/*.md` | Markdown rule files for AI behavior[^3] |
    | Project skills | `<project>/.trae/skills/` | Installed skill directories containing `SKILL.md` files[^4] |
    | Skill config | `<project>/.trae/skill-config.json` | Tracks disabled project skills[^4] |
    | AI ignore | `<project>/.trae/.ignore` | Files/folders to exclude from AI context[^7] |
    | Global skills | `~/.trae-cn/skills/` | Skills available across all projects (CN variant path)[^8] |
    | User rules | Settings UI > Rules | Personal rules applied to all projects[^3] |
  </Tab>
</Tabs>

### Project Directory Structure

```
.trae/
├── mcp.json              # MCP server configurations
├── skill-config.json     # Skill activation toggles
├── .ignore               # Files excluded from AI context
├── rules/                # Project rule files
│   └── project_rules.md
└── skills/               # Installed skills
    └── skill-name/
        └── SKILL.md
```

### Compatible Configuration Files

Trae can import rules from other AI coding tools. The following files are recognized when present in a project:[^3]

- `AGENTS.md`
- `CLAUDE.md`
- `CLAUDE.local.md`

## Custom Instructions / Rules

Trae supports two levels of rules for customizing AI behavior: **User Rules** (personal, all projects) and **Project Rules** (per-project, in `.trae/rules/`).[^3]

### User Rules

User rules are configured via **Settings > Rules & Skills**. Clicking **+ Create `user_rules.md`** generates a personal rules file. These rules apply across all projects and are useful for preferences like language, code style, or response format.[^9]

### Project Rules (`.trae/rules/*.md`)

Project rules are Markdown files stored in `.trae/rules/` with YAML frontmatter controlling activation behavior.[^3]

<CodeGroup>

```markdown Always Rule (.trae/rules/project_rules.md)
---
alwaysApply: true
---

Use TypeScript strict mode in all files.
Prefer functional components with hooks.
All public APIs must have JSDoc comments.
```

```markdown File-Pattern Rule (.trae/rules/api_standards.md)
---
alwaysApply: false
globs: "src/api/**/*.ts"
---

Use Zod for request/response validation.
All endpoints must return typed responses.
```

```markdown Intelligent Rule (.trae/rules/testing.md)
---
alwaysApply: false
description: "Testing conventions and patterns"
---

Use Vitest for all unit tests.
Mock external services at the boundary.
Aim for integration tests over unit tests where possible.
```

```markdown Manual Rule (.trae/rules/deployment.md)
---
alwaysApply: false
---

Deploy to staging first. Run smoke tests before production.
Use blue-green deployment strategy.
```

</CodeGroup>

### Rule Activation Modes

The rule type is determined by the combination of frontmatter fields:[^3]

| Mode | `alwaysApply` | `globs` | `description` | Behavior |
|---|---|---|---|---|
| **Always** | `true` | -- | -- | Always included in context for every request |
| **File-Pattern** | `false` | defined | -- | Included when referenced files match the glob pattern |
| **Intelligent** | `false` | -- | defined | AI determines relevance based on the description |
| **Manual** | `false` | -- | -- | Only included when explicitly referenced via `#RuleName` in chat |

<Note>
  Rules are loaded in full at the start of a chat session. For large instruction sets that should only load on-demand, use Skills instead of Rules to reduce token consumption.[^4]
</Note>

## Skills

Skills are reusable, scenario-specific capability sets defined through `SKILL.md` files. Unlike rules, skills use **on-demand loading** -- agents scan brief descriptions and only inject full content when relevant to the current task.[^4]

### Skill Structure

Each skill is a directory containing a required `SKILL.md` file and optional supporting resources:[^4]

```
skill-name/
├── SKILL.md        # Required: skill definition with YAML frontmatter
├── examples/       # Optional: example files
├── templates/      # Optional: template files
└── resources/      # Optional: additional resources
```

### SKILL.md Format

```markdown
---
name: my-skill
description: What this skill does and when to use it
---

# My Skill

## When to Use
Describe the conditions under which the agent should activate this skill.

## Instructions
Step-by-step instructions for the agent to follow.

## Examples
Demonstrate expected input/output behavior.
```

### Skill Scopes

| Scope | Path (Trae) | Path (Trae CN) |
|---|---|---|
| **Project** | `.trae/skills/{skill_name}/` | `.trae/skills/{skill_name}/` |
| **Global** | `~/.trae/skills/` | `~/.trae-cn/skills/` |

### Creating Skills

<Steps>
  <Step title="AI-Assisted (Recommended)">
    Describe the desired skill in chat and the AI generates the `SKILL.md` file automatically.[^4]
  </Step>
  <Step title="Manual">
    Open **Settings > Rules & Skills**, fill in the skill name, description, and instructions, then click Create.[^4]
  </Step>
  <Step title="Import">
    Upload a `SKILL.md` file or `.zip` archive containing a skill directory through the Settings UI.[^4]
  </Step>
  <Step title="CLI (npx skills)">
    Install community skills using the `npx skills` CLI:[^8]

    ```bash
    # Install a skill to Trae (project-level)
    npx skills add vercel-labs/agent-skills -a trae

    # Install a skill to Trae CN (project-level)
    npx skills add vercel-labs/agent-skills -a trae-cn

    # Install globally
    npx skills add vercel-labs/agent-skills -a trae --global
    ```
  </Step>
</Steps>

### Skill Configuration

A `skill-config.json` file in `.trae/` tracks which project skills are disabled, enabling granular control over skill activation per project.[^4]

## Custom Agents

Trae supports creating custom agents with configurable prompts, tool access, and MCP server scoping.[^10]

### Agent Properties

| Property | Description |
|---|---|
| **Name** | Required identifier for the agent |
| **Avatar** | Optional visual representation |
| **Prompt** | System instructions defining persona, response style, workflow, and operational rules |
| **Tools** | Selectable built-in capabilities (Read, Edit, Terminal, Preview, Web search) and MCP servers |
| **Callable by Other Agents** | Optional flag enabling modular agent composition with independent context |

### Creating Custom Agents

<Tabs>
  <Tab title="Smart Generation">
    Describe the agent's functionality and use cases in chat. Trae automatically generates the configuration based on the description.[^10]
  </Tab>
  <Tab title="Manual Creation">
    Configure agent parameters (name, prompt, tools, MCP access) sequentially through the Settings UI, then click Create.[^10]
  </Tab>
</Tabs>

Agents can be shared via platform links or imported from shared links.[^10]

<Note>
  Agent configurations are stored within Trae's internal database rather than in accessible file-system directories. They are not currently exportable as configuration files.[^10]
</Note>

## MCP Servers

Trae supports the Model Context Protocol for extending agent capabilities with external tools and data sources. MCP configuration is stored in `.trae/mcp.json`.[^2][^11]

### Configuration Format

<CodeGroup>

```json stdio Server (.trae/mcp.json)
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/path/to/dir"],
      "env": {
        "SOME_API_KEY": "value"
      }
    }
  }
}
```

```json HTTP Server (.trae/mcp.json)
{
  "mcpServers": {
    "remote-server": {
      "url": "https://mcp.example.com/sse",
      "headers": {
        "Authorization": "Bearer token"
      }
    }
  }
}
```

</CodeGroup>

### Transport Types

| Type | Configuration | Description |
|---|---|---|
| **stdio** | `command` + `args` + `env` | Local servers communicating via standard I/O using JSON-RPC 2.0[^11] |
| **SSE** | `url` + optional `headers` | HTTP-based transport with server streaming capabilities[^11] |
| **Streamable HTTP** | `url` + optional `headers` | Independent process handling multiple client connections with optional SSE[^11] |

### Adding MCP Servers

<Tabs>
  <Tab title="Marketplace">
    Navigate to **Settings > MCP > Add > Add from Marketplace**. Search for the desired server and click the **+** button to install.[^2]
  </Tab>
  <Tab title="Manual">
    Navigate to **Settings > MCP > Add > Add Manually**. Enter the JSON configuration and click Confirm.[^2]
  </Tab>
  <Tab title="Project Config File">
    Create or edit `.trae/mcp.json` in your project root. Enable **"Enable Project MCP"** in Settings to activate loading from this file.[^2]
  </Tab>
</Tabs>

### Advanced Configuration

- **Timeout**: Configure via `START_MCP_TIMEOUT_MS` and `RUN_MCP_TIMEOUT_MS` environment variables[^2]
- **Workspace variables**: Use `${workspaceFolder}` placeholder in args to reference the project root path[^2]
- **Auto-run**: MCP tool execution can be configured to run automatically or require confirmation via Settings[^12]

<Warning>
  MCP servers are built and maintained by third parties. Trae does not review or endorse these servers and is not responsible for their behavior.[^11]
</Warning>

## Hooks / Lifecycle Events

Trae does not currently expose a documented hooks or lifecycle events system for third-party extensions. Since Trae is built on the VS Code architecture, it inherits the standard VS Code extension activation/deactivation lifecycle for extensions installed via VSIX.[^13]

---

## VS Code Extension Compatibility

Trae supports VS Code extensions through three installation methods:[^13]

1. **Trae Extension Store** -- search and install from the built-in store
2. **VS Code Marketplace** -- download `.vsix` files from the VS Code marketplace and drag into Trae
3. **Local VSIX** -- install from local `.vsix` files via the extension store menu

Trae can also import extensions, settings, and key bindings from VS Code or Cursor during initial setup.[^12]

## Installation Provider Notes

This section covers what an AIPkg installer needs to know to programmatically install and configure packages for Trae and Trae CN.

### Configuration File Paths

<Tabs>
  <Tab title="Trae (International)">
    - **Project MCP config**: `<project>/.trae/mcp.json`
    - **Project rules**: `<project>/.trae/rules/*.md`
    - **Project skills**: `<project>/.trae/skills/{skill_name}/SKILL.md`
    - **Skill config**: `<project>/.trae/skill-config.json`
    - **Global skills**: `~/.trae/skills/`
  </Tab>
  <Tab title="Trae CN">
    - **Project MCP config**: `<project>/.trae/mcp.json`
    - **Project rules**: `<project>/.trae/rules/*.md`
    - **Project skills**: `<project>/.trae/skills/{skill_name}/SKILL.md`
    - **Skill config**: `<project>/.trae/skill-config.json`
    - **Global skills**: `~/.trae-cn/skills/`
  </Tab>
</Tabs>

### Installing MCP Server Extensions

To programmatically register an MCP server, merge the server definition into the `mcpServers` object in `.trae/mcp.json`:

```json
{
  "mcpServers": {
    "package-server-name": {
      "command": "npx",
      "args": ["-y", "@scope/mcp-server-package"],
      "env": {
        "SOME_API_KEY": "value"
      }
    }
  }
}
```

**Key considerations:**

- The `mcpServers` object is a flat map; each key is a unique server identifier
- Merge carefully -- do not overwrite existing server entries
- The root key **must** be `mcpServers`
- For stdio servers, use `command` + `args` + `env`; for HTTP servers, use `url` + optional `headers`
- Always include `-y` in npx args to avoid interactive prompts
- The user must enable **"Enable Project MCP"** in Settings for project-level `.trae/mcp.json` to be loaded[^2]
- MCP configuration is project-level only; there is no documented global `mcp.json` path

### Installing Skills

To install a skill, create a directory under `.trae/skills/` containing a `SKILL.md` file:

<Steps>
  <Step title="Create the skill directory">
    ```bash
    mkdir -p .trae/skills/my-package-skill
    ```
  </Step>
  <Step title="Write the SKILL.md file">
    ```markdown
    ---
    name: my-package-skill
    description: Guidelines and patterns for using my-package
    ---

    # My Package Skill

    ## When to Use
    When working with my-package in this project.

    ## Instructions
    Package-specific instructions here.
    ```
  </Step>
</Steps>

**Key considerations:**

- Project skills go in `<project>/.trae/skills/{skill_name}/SKILL.md`
- Global skills go in `~/.trae/skills/` (international) or `~/.trae-cn/skills/` (CN variant)[^8]
- Skills are loaded on-demand, so they do not consume context tokens until activated[^4]
- The `npx skills` CLI can also be used to install skills and will handle symlink creation automatically[^8]
- The `skill-config.json` file tracks disabled skills; do not remove entries from it when installing

### Installing Project Rules

To install project-level instructions, create Markdown files in `.trae/rules/`:

```markdown
---
alwaysApply: true
---

Instructions for using this package...
```

**Key considerations:**

- Create the `.trae/rules/` directory if it does not exist
- Use `alwaysApply: true` for rules that should always be in context
- Use `globs` for rules that apply only to specific file patterns
- Rules files can be version-controlled and committed to the repository
- Rules are loaded in full at chat start, so keep them concise to minimize token usage

### Trae vs Trae CN Installer Differences

The only difference relevant to installers is the **global skills path**:

| Variant | Global Skills Path |
|---|---|
| Trae | `~/.trae/skills/` |
| Trae CN | `~/.trae-cn/skills/` |

All project-level paths (`.trae/mcp.json`, `.trae/rules/`, `.trae/skills/`) are identical between variants.[^8]
