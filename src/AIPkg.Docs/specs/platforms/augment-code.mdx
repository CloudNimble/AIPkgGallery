## Overview

[Augment Code](https://www.augmentcode.com/) is an AI-powered coding agent developed by **Augment** (formerly Augment AI), backed by $252 million in funding from investors including former Google CEO Eric Schmidt. It is designed for professional software engineers working with large, complex codebases. Augment Code is available as extensions for **VS Code**, **JetBrains IDEs** (IntelliJ, PyCharm, WebStorm, etc.), **Vim/Neovim**, and as a **CLI tool** called **Auggie**.

Augment Code's key differentiator is its proprietary **Context Engine**, which maintains a live understanding of your entire stack -- code, dependencies, architecture, and commit history -- across 400,000+ files and multiple repositories. The platform supports the [Model Context Protocol (MCP)](https://modelcontextprotocol.io/) for extensibility.

<Info>
  Augment Code uses a multi-model architecture. It currently offers a model picker supporting models from Anthropic (Claude Sonnet) and OpenAI (GPT-5), with Augment handling the model hosting and routing. Users do not bring their own API keys -- models are accessed through the Augment platform.
</Info>

### Key Features

| Feature | Description |
|---|---|
| **Context Engine** | Indexes entire codebases (400K+ files) in real time, including cross-repo dependencies, commit history, and architectural patterns |
| **IDE Agents** | Autonomous agents that plan, implement, edit files, run terminal commands, and iterate on tasks within your IDE |
| **Remote Agents** | Cloud-based agents that work autonomously in the background, delivering completed PRs even when your IDE is closed |
| **Auggie CLI** | Full-featured terminal agent with the same Context Engine, supporting interactive and non-interactive (automation) modes |
| **MCP Support** | Connects to external tools and data sources via Model Context Protocol servers, plus 100+ native integrations |
| **AI Code Review** | Context-powered code reviews that catch critical bugs with high precision and recall |
| **Multi-Repo Understanding** | Seamless reasoning across multiple repositories and their dependencies |
| **Cross-Session Memory** | Agents remember context across sessions, allowing you to pick up complex tasks where you left off |
| **Augment Rules** | Project-level and user-level instruction files that guide agent behavior |
| **Checkpoints** | Automatic snapshots of workspace state at each agent step, enabling easy rollback |

### MCP Server Configuration

<Tabs>
  <Tab title="VS Code (settings.json)">
    MCP servers are configured in VS Code's `settings.json` under the `augment.advanced` key. The `mcpServers` field uses an **array** format:

    ```json
    {
      "augment.advanced": {
        "mcpServers": [
          {
            "name": "my-mcp-server",
            "command": "npx",
            "args": ["-y", "@scope/mcp-server-package"],
            "env": {
              "API_KEY": "your-key-here"
            }
          },
          {
            "name": "another-server",
            "command": "node",
            "args": ["./path/to/server.js"]
          }
        ]
      }
    }
    ```

    <Note>
      The VS Code `augment.advanced.mcpServers` uses an **array** format where each entry has a `name` field. This differs from the object-keyed format used by some other tools. The `${workspaceFolder}` variable is supported for dynamic path expansion.
    </Note>
  </Tab>
  <Tab title="Auggie CLI (settings.json)">
    MCP servers are configured in `~/.augment/settings.json` using an **object-keyed** format:

    ```json
    {
      "mcpServers": {
        "my-mcp-server": {
          "command": "npx",
          "args": ["-y", "@scope/mcp-server-package"],
          "env": {
            "API_KEY": "your-key-here"
          }
        }
      }
    }
    ```

    You can also manage MCP servers via CLI commands:

    ```bash
    # Add an MCP server
    auggie mcp add my-server --command npx --args "-y @scope/mcp-package" --env API_KEY=value

    # List configured MCP servers
    auggie mcp list

    # Remove an MCP server
    auggie mcp remove my-server

    # Check MCP server connectivity (in interactive mode)
    /status
    ```
  </Tab>
  <Tab title="Import from JSON">
    In the Augment Settings Panel, you can import MCP servers from a JSON configuration. The import format uses the **object-keyed** structure:

    ```json
    {
      "mcpServers": {
        "server-name": {
          "command": "npx",
          "args": ["-y", "@scope/mcp-server-package"],
          "env": {
            "API_TOKEN": "your-token-here"
          }
        }
      }
    }
    ```
  </Tab>
</Tabs>

### Easy MCP

Augment Code offers **Easy MCP**, a one-click integration system for popular developer tools. Instead of manually configuring servers, you can connect tools like CircleCI, MongoDB, Redis, Sentry, Stripe, and others with just an API token or OAuth approval. Access Easy MCP from the Augment Settings Panel.

## Custom Instructions

Augment Code supports several layers of custom instructions, called **Rules & Guidelines**.

### Augment Rules (Recommended)

Rules are Markdown files that live in the `.augment/rules/` directory. Three types of rules are supported:

| Type | Behavior |
|---|---|
| **Always** | Contents are included in every user prompt automatically |
| **Manual** | Must be explicitly attached via `@` mention in the chat |
| **Auto** | Agent automatically detects and attaches rules based on a description field in the frontmatter |

```markdown .augment/rules/coding-standards.md
---
type: always
---

# Coding Standards

- Use TypeScript strict mode with no `any` types
- Prefer functional patterns over classes
- All public APIs must have JSDoc comments
- Write unit tests for all new functions
```

**Placement locations:**

| Scope | Location |
|---|---|
| Project-wide | `<project-root>/.augment/rules/` |
| User-global | `~/.augment/rules/` |

<Info>
  User rules stored in `~/.augment/rules/` are always treated as the **Always** type and are automatically included in every prompt, regardless of any frontmatter configuration.
</Info>

### User Guidelines (IDE Settings)

User Guidelines are defined in the Augment Settings Panel within your IDE. They are stored locally and apply to all future chats in that IDE instance. Guidelines defined in VS Code do not propagate to JetBrains and vice versa.

### Legacy Workspace Guidelines

The `.augment-guidelines` file at the project root is a legacy mechanism. Its contents are included in all Agent and Chat sessions on that codebase. Augment will automatically import legacy guidelines as Rules.

```text .augment-guidelines
This is a TypeScript monorepo using pnpm workspaces.

To run tests: pnpm test
To build: pnpm build

Key modules:
- packages/core: Core library
- packages/cli: Command-line interface
- apps/web: Next.js web application
```

### Character Limits

| Scope | Limit |
|---|---|
| User Guidelines | 24,576 characters |
| Workspace Guidelines + Rules | 49,512 characters |

<Warning>
  Keep rules and guidelines concise and focused. The combined workspace guidelines and rules are limited to approximately 49,500 characters. Use Auto-type rules to conditionally include detailed standards only when relevant.
</Warning>

## Approval / Autonomy Modes

Augment Code provides a spectrum of autonomy settings for controlling agent behavior:

| Mode | Behavior |
|---|---|
| **Quick Ask** | Read-only mode. Agent can only read and analyze code; no file edits or command execution. Toggle via the Quick Ask button in the chat interface. |
| **Non-Auto** (default) | Agent pauses before executing terminal commands or accessing external integrations. Click the play button to approve each action. |
| **Auto** | Fully autonomous execution. Agent plans, implements, and iterates without pausing for approval. File edits, terminal commands, and integration calls all execute automatically. |

### Checkpoints

Regardless of the autonomy mode, Augment Code automatically saves **checkpoints** -- snapshots of your workspace at each agent step. You can revert to any previous checkpoint using the reverse arrow icon, providing a safety net for autonomous execution.

### CLI Permissions

In the Auggie CLI, the permission system provides an interactive approval workflow with allow/deny/ask rules:

```bash
# Configure permissions via flag
auggie --permission <level>

# Disable specific tools for a session
auggie --remove-tool web-fetch --remove-tool web-search
```

## Tools and Capabilities

### Built-in Tools

| Tool | Description |
|---|---|
| **File Operations** | Create, edit, and delete files across the workspace with full diff visibility |
| **Terminal Commands** | Execute shell commands, read output, and react to errors automatically |

### MCP Server Support

Augment Code supports MCP servers for extending agent capabilities:

| Feature | Details |
|---|---|
| **Transport types** | stdio (local commands), SSE, and HTTP |
| **Configuration** | Via IDE settings panel, `settings.json`, CLI commands, or Easy MCP |

| **Native integrations** | 100+ built-in integrations including Jira, Confluence, GitHub, CircleCI, Sentry, and more |

### Remote Agents

Remote Agents run in cloud environments, working autonomously in the background:

- Execute tasks asynchronously (bug fixes, refactoring, test generation, documentation)
- Produce review-ready pull requests
- Run up to 10 agents in parallel
- Monitor progress in real-time; connect via SSH when needed
- Continue working even when your IDE is closed

### Parallel Tool Execution

Augment Code automatically parallelizes file reads, searches, and API calls, enabling up to 2x faster turns on complex tasks that touch many files.

## Model Configuration

### Available Models

Augment Code hosts and routes models through its own platform. Users select from available models via a model picker rather than providing their own API keys.

| Model | Notes |
|---|---|
| **GPT-5.4** | Current default model. Excels at multi-agent orchestration and verification. |
| **Claude Sonnet 4.5** | Faster, more direct responses. Previously the default model. |
| **GPT-5 / GPT-5.2** | Earlier GPT-5 variants available in the model picker. |

<Note>
  Augment Code does not support bring-your-own-key (BYOK) model configuration. All models are accessed through the Augment platform. The available models and defaults may change as Augment rolls out updates.
</Note>

### Switching Models

## CLI Commands (Auggie)

### Installation

```bash
npm install -g @augmentcode/auggie
```

<Note>
  Auggie requires **Node.js version 22 or higher**. The CLI is distributed as an npm package.
</Note>

### Core Commands

| Command | Description |
|---|---|
| `auggie` | Start an interactive terminal session |
| `auggie "prompt"` | Run a prompt in interactive mode |
| `auggie --print "prompt"` | Run a prompt, print result to stdout, and exit |
| `auggie --print --quiet "prompt"` | Minimal output suitable for piping and scripting |
| `auggie login` | Authenticate with the Augment service |
| `auggie --model <name>` | Start with a specific model |
| `auggie --rules <path>` | Specify rules file location |
| `auggie -w <path>` | Specify workspace directory |

### MCP Management

| Command | Description |
|---|---|
| `auggie mcp add <name>` | Add an MCP server to settings |
| `auggie mcp list` | List configured MCP servers |
| `auggie mcp remove <name>` | Remove an MCP server |
| `auggie --mcp` | Run Auggie as an MCP server (exposes codebase-retrieval tool) |
| `auggie --mcp-auto-workspace` | Enable dynamic workspace discovery in MCP mode |
| `auggie --mcp-config <path>` | Load MCP config from a file (overrides settings) |

### Tool Management

| Command | Description |
|---|---|
| `auggie tools list` | List available tools |
| `auggie tools add <name>` | Add a tool |
| `auggie tools remove <name>` | Remove a tool |
| `auggie --remove-tool <name>` | Disable a tool for the session |

### Custom Commands

| Command | Description |
|---|---|
| `auggie command list` | List available custom commands |
| `auggie command <name>` | Execute a custom command |
| `auggie /<command-name>` | Execute a custom command (shorthand) |
| `auggie /<command> -p` | Execute a custom command non-interactively |

### Other Flags

| Flag | Description |
|---|---|
| `--permission <level>` | Set permission/approval level |
| `--acp` | Run as an Agent Client Protocol server |
| `--allow-indexing` | Allow codebase indexing (used with `--acp`) |

### Interactive Slash Commands

Within an active Auggie session, these slash commands are available:

| Command | Description |
|---|---|
| `/ask` | Enter read-only ask mode (no code changes) |
| `/model` | Switch the active model |
| `/status` | Verify MCP server connectivity |
| `/rules` | View attached rules |
| `/feedback` | Send feedback or report bugs |
| `/github-workflow` | Generate GitHub Actions workflows |
| `/exit` | Exit the interactive CLI |

### Custom Slash Commands

Custom commands are Markdown files stored in command directories:

| Scope | Location |
|---|---|
| User commands | `~/.augment/commands/<name>.md` |
| Workspace commands | `.augment/commands/<name>.md` |
| Claude Code compatibility | `.claude/commands/<name>.md` |

Commands in nested directories use colon-separated namespacing (e.g., `.augment/commands/frontend/component.md` becomes `/frontend:component`).

## Installation Provider Notes

This section covers what an AIPkg installer needs to know to programmatically install and configure packages for the Augment Code platform.

### Configuration File Paths

<Tabs>
  <Tab title="VS Code">
    - **MCP servers**: `.vscode/settings.json` (under `augment.advanced.mcpServers`)
    - **Workspace rules**: `<project>/.augment/rules/`
    - **Legacy guidelines**: `<project>/.augment-guidelines`
    - **Ignore file**: `<project>/.augmentignore`
  </Tab>
  <Tab title="Auggie CLI">
    - **Primary config**: `~/.augment/settings.json`
    - **User rules**: `~/.augment/rules/`
    - **User commands**: `~/.augment/commands/`
    - **Workspace rules**: `<project>/.augment/rules/`
    - **Workspace commands**: `<project>/.augment/commands/`
    - **Legacy guidelines**: `<project>/.augment-guidelines`
  </Tab>
</Tabs>

### Installing MCP Server Extensions

To programmatically register an MCP server, the approach differs between VS Code and CLI.

<Tabs>
  <Tab title="VS Code">
    Merge the MCP server definition into `.vscode/settings.json` under the `augment.advanced.mcpServers` array:

    ```json
    {
      "augment.advanced": {
        "mcpServers": [
          {
            "name": "package-server-name",
            "command": "npx",
            "args": ["-y", "@scope/mcp-server-package"],
            "env": {
              "SOME_API_KEY": "value"
            }
          }
        ]
      }
    }
    ```

    **Key considerations:**
    - The `mcpServers` field is an **array**, not an object map -- each entry is identified by its `name` field
    - When merging, check existing entries by `name` to avoid duplicates
    - The `${workspaceFolder}` variable is supported for dynamic path expansion
    - Restart VS Code or reload the window after modifying MCP server configuration
  </Tab>
  <Tab title="Auggie CLI">
    Merge the MCP server definition into `~/.augment/settings.json` under the `mcpServers` object:

    ```json
    {
      "mcpServers": {
        "package-server-name": {
          "command": "npx",
          "args": ["-y", "@scope/mcp-server-package"],
          "env": {
            "SOME_API_KEY": "value"
          }
        }
      }
    }
    ```

    Alternatively, use the CLI command:

    ```bash
    auggie mcp add package-server-name --command npx --args "-y @scope/mcp-server-package" --env SOME_API_KEY=value
    ```

    **Key considerations:**
    - The CLI `mcpServers` field is an **object map** keyed by server name
    - Merge carefully -- do not overwrite existing entries
    - The CLI also supports `type`, `url`, and `headers` fields for remote MCP servers
  </Tab>
</Tabs>

<Warning>
  The VS Code extension and Auggie CLI use **different JSON formats** for MCP server configuration. The VS Code extension uses an array with `name` fields, while the CLI uses an object map keyed by server name. An installer must handle both formats appropriately.
</Warning>

### Installing Project Instructions

To install project-level instructions, create a rules file in the `.augment/rules/` directory:

```bash
mkdir -p .augment/rules
cat > .augment/rules/my-package-rules.md << 'EOF'
---
type: always
---

# My Package Instructions

These instructions guide the agent when working with this codebase.
EOF
```

Alternatively, append to the legacy `.augment-guidelines` file at the project root if one already exists.

<Note>
  The `.augment/rules/` directory is the recommended approach. Rules files are Markdown with optional YAML frontmatter specifying the type (`always`, `manual`, or `auto`). Multiple rules files can coexist in the directory.
</Note>

### Config File Format

Augment Code uses JSON for configuration files:

- **VS Code**: Standard VS Code `settings.json` format. The installer must parse existing JSON, merge into the `augment.advanced.mcpServers` array, and write back.
- **CLI**: `~/.augment/settings.json` is a JSON file. The installer must parse existing JSON, deep-merge the `mcpServers` object, and write back preserving existing entries.

### Authentication

Augment Code uses its own authentication system (account-based login). There are no user-provided API keys for model access. Authentication is handled via `auggie login` (CLI) or the sign-in flow in the IDE extension.

### Non-Interactive Execution

For CI/CD or automated workflows using the CLI:

```bash
# Run a prompt and print the result
auggie --print "Run the test suite and report results"

# Minimal output for scripting
auggie --print --quiet "Check if all new functions have tests"

# Execute a custom command
auggie command code-review

# Disable specific tools for safety
auggie --remove-tool web-fetch --print "Analyze this codebase"
```
