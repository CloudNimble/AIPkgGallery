## Overview

[Aider](https://aider.chat/) is an open-source AI pair programming tool created by **Paul Gauthier** that runs directly in your terminal. Licensed under Apache 2.0 and hosted at [Aider-AI/aider](https://github.com/Aider-AI/aider) on GitHub (40k+ stars), Aider connects to your local Git repository and lets you collaboratively edit code with LLMs using natural language. It automatically commits every change with descriptive messages, making AI-assisted edits fully reversible with standard Git tooling.

Aider is built on a unique **repository map** system powered by tree-sitter. It constructs a graph of your entire codebase -- classes, functions, and their relationships -- and uses PageRank to select the most relevant context for each request. This allows it to work effectively across large, multi-file projects without exceeding token limits.

<Info>
  Aider is model-agnostic. It connects to virtually any LLM via the [LiteLLM](https://github.com/BerriAI/litellm) library, including Anthropic Claude, OpenAI GPT/o-series, Google Gemini, DeepSeek, xAI Grok, Ollama (local), OpenRouter, AWS Bedrock, Azure OpenAI, and many more.
</Info>

### Key Features

| Feature | Description |
|---|---|
| **Multi-File Editing** | Coordinates changes across multiple files in a single request with automatic conflict resolution |
| **Git-Native Workflow** | Automatically commits every AI change with descriptive messages; `/undo` instantly reverts the last commit |
| **Repository Map** | Tree-sitter-powered map of your entire codebase, ranked by relevance using PageRank for optimal context |
| **Chat Modes** | Four modes -- code, ask, architect, and help -- for different interaction patterns |
| **Architect Mode** | Two-model pipeline: an architect model plans changes, an editor model applies them |
| **Linting & Testing** | Automatically runs linters and tests after changes, then fixes detected issues |
| **Voice Input** | Speak requests directly via `/voice`; supports transcription to text |
| **Web Scraping** | Pull web pages into chat context with `/web <url>` for reference documentation |
| **100+ Languages** | Works with Python, JavaScript, TypeScript, Rust, Go, C++, Ruby, PHP, HTML, CSS, and dozens more |
| **Conventions Files** | Load project-specific coding guidelines as read-only context via `CONVENTIONS.md` |

## Configuration

### File Locations

Aider uses a layered configuration system. Settings can be specified via command-line flags, YAML config files, environment variables, or `.env` files. Later sources override earlier ones.

| File | Search Locations | Purpose |
|---|---|---|
| `.aider.conf.yml` | `~/.aider.conf.yml`, `<git-root>/.aider.conf.yml`, `<cwd>/.aider.conf.yml` | Primary YAML configuration (all settings) |
| `.env` | `~/.env`, `<git-root>/.env`, `<cwd>/.env` | Environment variables and API keys for all providers |
| `.aider.input.history` | `<git-root>/` | Chat input history |
| `.aider.chat.history.md` | `<git-root>/` | Markdown log of chat history |
| `.aider.tags.cache.v3/` | `<git-root>/` | Disk cache for tree-sitter repository map tags |
| `.aiderignore` | `<git-root>/` | Gitignore-format file to exclude files from aider's view |
| `.aider.model.settings.yml` | Specified via `--model-settings-file` | Custom model settings (edit formats, provider routing) |

<Note>
  Files loaded later in the search order take priority. For example, a `.aider.conf.yml` in the current directory overrides the one in your home directory. You can also specify a custom config path with `--config <filename>` or a custom env file with `--env-file <filename>`.
</Note>

### Configuration Priority

Settings are resolved in the following order (highest to lowest):

1. **Command-line flags** (e.g., `--model anthropic/claude-sonnet-4-20250514`)
2. **Environment variables** (e.g., `AIDER_MODEL=anthropic/claude-sonnet-4-20250514`)
3. **`.env` file** in current directory
4. **`.aider.conf.yml`** in current directory
5. **`.env` file** in git root
6. **`.aider.conf.yml`** in git root
7. **`.env` file** in home directory
8. **`.aider.conf.yml`** in home directory
9. **Default values**

### The Conventions File

Aider supports project-specific coding conventions via a `CONVENTIONS.md` (or any named file) loaded as read-only context. This guides the LLM to follow your coding style, testing practices, and documentation standards.

```markdown CONVENTIONS.md
# Coding Conventions

## Style
- Use TypeScript strict mode with no `any` types
- Prefer functional patterns over classes
- All exported functions must have JSDoc comments

## Testing
- Write tests using vitest
- Aim for 80%+ coverage on new code
- Use descriptive test names: "should [expected behavior] when [condition]"

## Git
- Use conventional commits (feat:, fix:, chore:, etc.)
- Keep commits atomic and focused

## Architecture
- Follow hexagonal architecture patterns
- Keep business logic in the domain layer
- Use dependency injection for external services
```

**Loading conventions:**

| Method | Usage |
|---|---|
| CLI flag | `aider --read CONVENTIONS.md` |
| In-chat command | `/read-only CONVENTIONS.md` |
| Config file | `read: CONVENTIONS.md` in `.aider.conf.yml` |

<Warning>
  Load conventions files with `--read` (read-only mode), not `--file`. Read-only files benefit from prompt caching when `cache-prompts: true` is enabled, reducing costs for repeated sessions. They also cannot be accidentally edited by the LLM.
</Warning>

## Chat Modes

Aider provides four chat modes that control how the LLM interacts with your code. You can switch modes mid-conversation using slash commands.

| Mode | Command | Description |
|---|---|---|
| **Code** | `/code` | Default mode. The LLM edits files directly based on your requests. |
| **Ask** | `/ask` | Question-only mode. The LLM answers questions about the codebase without making any edits. |
| **Architect** | `/architect` | Two-model pipeline. The main model acts as an architect to plan changes; a separate editor model translates the plan into file edits. |
| **Help** | `/help` | Ask questions about aider itself -- features, settings, troubleshooting, and LLM usage. |

### Switching Modes

- **Sticky switch**: A bare `/ask`, `/code`, or `/architect` command (with no message) permanently changes the active mode.
- **One-shot**: `/ask how does the auth module work?` sends that single message in ask mode, then returns to the previous mode.
- **Explicit mode**: `/chat-mode <mode>` sets the mode permanently.

### Architect Mode Details

Architect mode is particularly useful for models that struggle to both plan and edit in a single pass. The architect model proposes a solution, and the editor model generates precise file edits.

- Configure the architect model with `--model <model>`
- Configure the editor model with `--editor-model <model>`
- The recommended editor edit formats are `editor-diff` and `editor-whole`
- Use `--auto-accept-architect true` to skip confirmation of architect proposals

<Info>
  A common workflow is to alternate between `/ask` and `/code` manually, treating it as a fluid version of architect mode with a single model. Use ask mode to discuss the approach, then switch to code mode for implementation.
</Info>

## MCP Support

Aider does not natively support the Model Context Protocol (MCP) as of early 2026. There is an [open feature request](https://github.com/aider-ai/aider/issues/4506) to add native MCP server and agent mode support.

However, several third-party projects bridge Aider with the MCP ecosystem:

| Project | Description |
|---|---|
| **[aider-mcp-server](https://github.com/disler/aider-mcp-server)** | MCP server that exposes Aider as a tool, allowing Claude Code or other MCP clients to delegate coding tasks to Aider |
| **[AiderDesk](https://github.com/hotovo/aider-desk)** | Desktop GUI built on Aider with full MCP server support, agent mode, and built-in power tools |
| **mcpm-aider** | General-purpose management tool for connecting Aider to any MCP server and its tools |

<Note>
  While Aider itself cannot consume MCP servers, it can be exposed *as* an MCP server via third-party wrappers. This allows MCP-native clients (like Claude Desktop or Goose) to offload coding tasks to Aider.
</Note>

## Tools and Capabilities

### Git Integration

Aider is deeply integrated with Git. Every successful edit is automatically committed with a descriptive AI-generated message.

| Feature | Description |
|---|---|
| **Auto-commit** | Each AI edit is committed automatically (disable with `--no-auto-commits`) |
| **Dirty repo handling** | Uncommitted changes are committed first to keep AI and human work separate |
| **Undo** | `/undo` reverts the last AI commit instantly |
| **Diff** | `/diff` shows changes since the last message |
| **Attribution** | Commits are marked with `aider:` prefix and `Co-authored-by` trailers |
| **Commit messages** | Generated by a separate "weak model" to save costs |
| **Pre-commit hooks** | Controlled via `git-commit-verify` setting |

### Repository Map

The repository map is one of Aider's most distinctive technical features:

1. **Tree-sitter parsing** extracts symbol definitions (classes, functions, methods) from all files in the repo
2. **Graph construction** builds a directed graph where files are nodes and dependencies are edges
3. **PageRank ranking** identifies the most relevant files based on what is currently in the chat
4. **Token-budgeted output** fits the most important context within a configurable token limit (`--map-tokens`)

The map allows the LLM to understand your codebase structure without loading every file into context.

### Linting and Testing

| Feature | Command | Description |
|---|---|---|
| **Auto-lint** | Automatic | Runs after each AI edit when `auto-lint: true` |
| **Manual lint** | `/lint` | Lint files in the chat and fix issues |
| **Auto-test** | Automatic | Runs after each AI edit when `auto-test: true` |
| **Manual test** | `/test` | Run tests and share failures with the LLM for fixing |
| **Custom commands** | `--lint-cmd`, `--test-cmd` | Specify per-language linters and test runners |

### Web Scraping

Use `/web <url>` to scrape a web page, convert it to markdown, and include it in the chat context. Useful for referencing documentation, API specs, or issue descriptions.

### Voice Input

Use `/voice` to record audio input, which is transcribed and sent as a chat message. Configurable via `--voice-format` (wav/mp3) and `--voice-language` (ISO 639-1 code).

### Configuring Models

<Steps>
  <Step title="Set the API key">
    Set your provider's API key as an environment variable, in a `.env` file, or via the `--api-key` flag:

    ```bash
    export ANTHROPIC_API_KEY=sk-ant-...
    # or
    aider --api-key anthropic=sk-ant-...
    ```
  </Step>
  <Step title="Select a model">
    Specify the model on the command line or in `.aider.conf.yml`:

    ```bash
    aider --model anthropic/claude-sonnet-4-20250514
    ```

    Use `aider --list-models <search>` to search available models. If no model is specified, Aider selects one based on available API keys.
  </Step>
  <Step title="Configure supporting models (optional)">
    Set a weak model for commit messages and a separate editor model for architect mode:

    ```bash
    aider --model anthropic/claude-sonnet-4-20250514 \
          --weak-model anthropic/claude-haiku-3-5-20241022 \
          --editor-model anthropic/claude-sonnet-4-20250514
    ```
  </Step>
</Steps>

### Multi-Model Configuration

Aider uses up to three models simultaneously:

| Role | Flag | Purpose |
|---|---|---|
| **Main model** | `--model` | Primary model for code generation and conversation |
| **Weak model** | `--weak-model` | Generates commit messages and summarizes chat history (cheaper, faster) |
| **Editor model** | `--editor-model` | Translates architect proposals into file edits (used in architect mode) |

### Custom Model Settings

For models not natively recognized by Aider, create a `.aider.model.settings.yml` file:

```yaml .aider.model.settings.yml
- name: openrouter/my-custom-model
  edit_format: diff
  weak_model_name: openrouter/my-cheap-model
  use_repo_map: true
  send_undo_reply: true
  accepts_images: false
  lazy: false
  reminder: sys
  examples_as_sys_msg: true
```

## Installation

<Tabs>
  <Tab title="One-liner (Recommended)">
    No Python required. This installs `uv`, Python 3.12, and aider in an isolated environment:

    ```bash
    curl -LsSf https://aider.chat/install.sh | sh
    ```
  </Tab>
  <Tab title="aider-install">
    Requires Python 3.8+. Installs only `uv` and `aider-install` in your Python environment, then uses `uv` to install aider in isolation:

    ```bash
    pip install aider-install && aider-install
    ```
  </Tab>
  <Tab title="uv (Direct)">
    If you already have `uv` installed:

    ```bash
    uv tool install --force --python python3.12 --with pip aider-chat@latest
    ```
  </Tab>
  <Tab title="pipx">
    ```bash
    pipx install aider-chat
    ```
  </Tab>
  <Tab title="pip">
    ```bash
    python -m pip install -U --upgrade-strategy only-if-needed aider-chat
    ```
  </Tab>
  <Tab title="Docker">
    ```bash
    # Core image
    docker pull paulgauthier/aider
    docker run -it --user $(id -u):$(id -g) \
      --volume $(pwd):/app \
      paulgauthier/aider \
      --anthropic-api-key $ANTHROPIC_API_KEY

    # Full image (includes Playwright, browser GUI support)
    docker pull paulgauthier/aider-full
    docker run -it --user $(id -u):$(id -g) \
      --volume $(pwd):/app \
      paulgauthier/aider-full \
      --anthropic-api-key $ANTHROPIC_API_KEY
    ```

    Run from the root of your git repo so the `--volume` flag maps the correct directory.
  </Tab>
</Tabs>

<Warning>
  System package managers (Homebrew, apt, etc.) often install aider with incorrect dependency versions. The recommended installation methods are the one-liner, `aider-install`, `uv`, or `pipx`, which all ensure proper dependency isolation.
</Warning>

Verify installation:

```bash
aider --version
```

## Installation Provider Notes

This section covers what an AIPkg installer needs to know to programmatically install and configure packages for the Aider platform.

### Configuration File Paths

| File | Path | Purpose |
|---|---|---|
| **YAML config** | `~/.aider.conf.yml` or `<git-root>/.aider.conf.yml` | Primary configuration (all settings) |
| **Environment file** | `~/.env` or `<git-root>/.env` | API keys for all providers, general settings as `AIDER_*` variables |
| **Conventions** | `<project>/CONVENTIONS.md` (or any name) | Read-only coding guidelines loaded via `read:` config |
| **Ignore file** | `<git-root>/.aiderignore` | Gitignore-format list of files to exclude from aider |
| **Model settings** | `<git-root>/.aider.model.settings.yml` | Custom model configurations and provider routing |

### Installing Custom Instructions

To install project-level instructions, write a conventions file (e.g., `CONVENTIONS.md`) to the project root and add it to the `read` key in `.aider.conf.yml`:

```yaml
read:
  - CONVENTIONS.md
```

If a `.aider.conf.yml` already exists, the installer must parse the existing YAML and append to the `read` list without overwriting other entries.

<Note>
  Unlike platforms with a single fixed instructions file, Aider supports loading multiple read-only files. The `read` key accepts a list, so appending new entries is safe and will not conflict with existing conventions files.
</Note>

### API Key Management

API keys can be configured in multiple ways:

1. **`.env` file** (recommended for installers): Write keys to `<git-root>/.env` or `~/.env`
   ```
   ANTHROPIC_API_KEY=sk-ant-...
   OPENAI_API_KEY=sk-...
   ```
2. **Environment variables**: Set in the user's shell profile
3. **YAML config**: Only `openai-api-key` and `anthropic-api-key` are supported in `.aider.conf.yml`
4. **CLI flag**: `--api-key provider=<key>` (sets the corresponding `PROVIDER_API_KEY` environment variable)

<Warning>
  The YAML config file only supports OpenAI and Anthropic API keys directly. All other provider keys must be stored in a `.env` file or set as environment variables. An installer targeting non-OpenAI/Anthropic providers should always use the `.env` file.
</Warning>

### Config File Format

Aider uses standard YAML for `.aider.conf.yml`. The installer must:

1. Parse the existing YAML file (if it exists)
2. Merge new settings, being careful with list-type values (`read`, `file`, `lint-cmd`) -- append rather than overwrite
3. Write the result back, preserving existing entries

Key settings an installer may need to modify:

| Setting | Type | Purpose |
|---|---|---|
| `model` | string | Set the default LLM |
| `read` | list | Add read-only context files (conventions, docs) |
| `auto-lint` | boolean | Enable/disable auto-linting |
| `lint-cmd` | list | Configure per-language linters |
| `test-cmd` | string | Configure the test runner |
| `auto-test` | boolean | Enable/disable auto-testing |
| `map-tokens` | integer | Token budget for repository map |

### Non-Interactive Execution

For CI/CD, automation, or scripting:

```bash
# Run a single instruction and exit
aider --message "Add error handling to the API endpoints" --yes-always

# Run instructions from a file
aider --message-file instructions.md --yes-always

# Process multiple files with the same instruction
for FILE in src/*.py; do
    aider --message "Add type hints to all functions" --yes-always "$FILE"
done

# Dry run (preview changes without applying)
aider --message "Refactor the database layer" --dry-run
```

**Key flags for scripting:**

| Flag | Environment Variable | Description |
|---|---|---|
| `--message` | `AIDER_MESSAGE` | Send a single message and exit |
| `--message-file` | `AIDER_MESSAGE_FILE` | Read message from a file and exit |
| `--yes-always` | `AIDER_YES_ALWAYS` | Auto-confirm all prompts |
| `--no-auto-commits` | `AIDER_AUTO_COMMITS=false` | Disable automatic git commits |
| `--dry-run` | `AIDER_DRY_RUN` | Preview without modifying files |
| `--no-git` | `AIDER_GIT=false` | Disable all git integration |
| `--read` | `AIDER_READ` | Add read-only context files |

### Detecting Aider Installation

An installer can check for aider with:

```bash
# Check if aider is installed
command -v aider >/dev/null 2>&1

# Check version
aider --version

# Check if running inside a git repo (required by aider)
git rev-parse --is-inside-work-tree >/dev/null 2>&1
```
